# The 21 IT-TRIZ Parameters

| ID | Name | One-line scope |
|----|------|----------------|
| 1 | Size (Static) | Compiled binary, lines of code, on-disk footprint |
| 2 | Size (Dynamic) | RAM, runtime heap, dynamic bandwidth |
| 3 | Amount of Data | Volume of information processed, transmitted, or stored |
| 4 | Interface | Boundary layers between components: APIs, ports, protocol translation |
| 5 | Speed | Throughput, request latency, algorithmic execution time |
| 6 | Accuracy | Precision of outputs, algorithmic correctness, floating-point fidelity |
| 7 | Stability | Crash resistance, uptime, absence of race conditions or deadlocks |
| 8 | Ability to Detect/Measure | Observability: telemetry, logging, error tracing, metric exposure |
| 9 | Loss of Time | I/O blocking, excessive compilation, synchronization delays |
| 10 | Loss of Data | Packet drops, memory leaks, database corruption, cache eviction |
| 11 | Harmful Effects Generated by System | CPU heat, memory hogging that starves other processes, API flooding |
| 12 | Adaptability / Versatility | Cross-platform support, modularity, responsiveness to environmental change |
| 13 | Compatibility / Connectability | Legacy architecture interface, external protocol conformance, interoperability |
| 14 | Ease of Use | Cognitive load on end-users and developers (GUI intuitiveness, API clarity) |
| 15 | Reliability / Robustness | Fault tolerance, graceful degradation, recovery from unexpected state |
| 16 | Security | Resilience against unauthorized access, cryptographic strength, auth controls |
| 17 | Aesthetics / Appearance | Visual presentation, frontend rendering quality, structural elegance |
| 18 | Harmful Effects on System | Susceptibility to external degradation: DDoS, SQL injection, hardware failure |
| 19 | System Complexity | Tight coupling, spaghetti code, deep inheritance trees, circular dependencies |
| 20 | Control Complexity | Overhead of configuration management, deployment pipelines, scaling mechanics |
| 21 | Automation | Self-healing routines, automated testing, CI/CD pipeline coverage |

---

## 1. Size (Static)

**Scope:** The physical footprint of the software at rest — lines of code, compiled binary size, and static storage constraints on disk.

**Software examples:**
- A monolithic application with 500k LOC vs. a focused microservice with 2k LOC
- Compiled binary bloat from unused library inclusion
- Docker image layer size growing with each dependency added

---

## 2. Size (Dynamic)

**Scope:** The resource footprint of the software during execution — RAM allocation, runtime heap usage, and dynamic bandwidth utilization.

**Software examples:**
- A service that allocates 4 GB of heap to cache a dataset vs. one that streams it
- Memory pressure from large object graphs held in-session
- Network bandwidth consumed by chatty APIs under load

---

## 3. Amount of Data

**Scope:** The sheer volume of information being processed, transmitted, or stored — database scalability and big data streams.

**Software examples:**
- A time-series database ingesting 1M events per second
- Batch ETL pipeline processing petabytes of log archives
- A search index that must re-index 100M documents on schema change

---

## 4. Interface

**Scope:** Boundary layers between distinct software components or hardware — APIs, network ports, and protocol translation points.

**Software examples:**
- A REST API surface that dozens of external clients depend on
- A gRPC contract between a service and its consumers
- An event schema on a Kafka topic consumed by multiple downstream teams

---

## 5. Speed

**Scope:** The velocity of operations — clock cycles, transaction throughput, request latency, and algorithmic execution time.

**Software examples:**
- P99 latency of a database query
- Throughput of a message queue consumer
- Time complexity of a sorting or search algorithm on large datasets

---

## 6. Accuracy

**Scope:** The precision of system outputs — algorithmic correctness, floating-point precision, and fidelity of data transformations.

**Software examples:**
- Floating-point rounding errors accumulating in financial calculations
- ML model prediction accuracy vs. inference speed trade-off
- Data transformation pipeline that silently truncates decimal precision

---

## 7. Stability

**Scope:** Operational consistency over time — crash resistance, uptime, absence of race conditions or deadlocks.

**Software examples:**
- A service that crashes under concurrent write load due to a race condition
- Memory leak that degrades performance over days until OOM kill
- Distributed lock contention causing periodic deadlocks in a job scheduler

---

## 8. Ability to Detect/Measure

**Scope:** Observability of the system — telemetry, logging, error tracing, diagnostic capabilities, and metric exposure.

**Software examples:**
- A service with no structured logging, making root-cause analysis guesswork
- Distributed tracing that correlates requests across 10 microservices
- Custom Prometheus metrics exposing queue depth and processing lag

---

## 9. Loss of Time

**Scope:** Inefficiencies resulting in temporal waste — I/O blocking, excessive compilation times, synchronization delays across distributed nodes.

**Software examples:**
- Synchronous HTTP calls between services adding hundreds of milliseconds per hop
- A build pipeline that takes 40 minutes due to uncached dependency resolution
- Distributed saga waiting on a slow downstream service before proceeding

---

## 10. Loss of Data

**Scope:** Degradation or destruction of state — packet drops, memory leaks, database corruption, cache eviction.

**Software examples:**
- At-most-once delivery semantics causing message loss under network partition
- LRU cache eviction discarding critical session state
- Partial write failure leaving a database in an inconsistent state

---

## 11. Harmful Effects Generated by System

**Scope:** Detrimental outputs created by the software — excessive CPU heat, memory hogging that starves other processes, downstream API flooding.

**Software examples:**
- A background analytics job consuming 100% CPU and starving the primary service
- A retry loop with no backoff triggering a thundering herd on a recovering service
- Noisy neighbour effect in a shared-tenant cloud environment

---

## 12. Adaptability / Versatility

**Scope:** Portability and flexibility of the codebase — cross-platform support, modularity, responsiveness to dynamic environmental changes.

**Software examples:**
- A CLI tool that runs on Linux, macOS, and Windows without modification
- A plugin architecture that lets users extend functionality without forking
- Feature flags that allow runtime behaviour changes without redeployment

---

## 13. Compatibility / Connectability

**Scope:** Ability to interface with legacy architectures, conform to external protocols, and interoperate within disparate ecosystems.

**Software examples:**
- A new service that must speak an existing SOAP protocol for legacy client compatibility
- A database driver that must support both PostgreSQL 12 and 15 APIs
- An SDK published for three languages from a single OpenAPI spec

---

## 14. Ease of Use

**Scope:** Cognitive load imposed on the human operator — end-user GUIs and developer-facing API intuitiveness.

**Software examples:**
- An API where common operations require 5+ chained calls vs. a single idiomatic call
- An error message that says "Error: 500" vs. one that names the field and suggests a fix
- Onboarding flow that requires reading 10 docs before making a first successful request

---

## 15. Reliability / Robustness

**Scope:** Fault tolerance — graceful degradation, exception handling, and recovery from unexpected state changes.

**Software examples:**
- A payment service that retries idempotently and falls back to queuing on downstream timeout
- Circuit breaker pattern preventing cascade failure across a service mesh
- A CLI that handles partial downloads and resumes rather than starting over

---

## 16. Security

**Scope:** Resilience against unauthorized access — cryptographic strength, vulnerability mitigation, and robust authentication controls.

**Software examples:**
- JWT token with short expiry vs. long-lived session cookie
- SQL parameterization preventing injection attacks
- mTLS between internal microservices preventing lateral movement after perimeter breach

---

## 17. Aesthetics / Appearance

**Scope:** Visual presentation layer — frontend rendering quality and structural elegance of code.

**Software examples:**
- A dashboard that renders data clearly vs. one that overwhelms with raw JSON
- A codebase with consistent naming conventions vs. one mixing camelCase and snake_case
- A CLI output that uses colour and spacing to guide reading vs. undifferentiated text streams

---

## 18. Harmful Effects on System

**Scope:** The software's susceptibility to external degradation — DDoS attacks, SQL injection, network outages, or hardware failures.

**Software examples:**
- An unprotected API endpoint that becomes a target for scraping
- A service with no rate limiting, downed by a misbehaving client
- A stateful service with no reconnect logic that permanently fails after a 30-second network blip

---

## 19. System Complexity

**Scope:** Structural entanglement of the architecture — tight coupling, spaghetti code, deep inheritance trees, circular dependencies.

**Software examples:**
- A monolith where changing a utility function requires touching 40 files
- A microservices graph with 15 synchronous dependencies per service
- Circular imports between modules preventing independent testing

---

## 20. Control Complexity

**Scope:** Difficulty of orchestrating the system's lifecycle — configuration management, deployment pipelines, and scaling mechanics.

**Software examples:**
- Kubernetes manifests that require manual coordination across 8 services to deploy one change
- A CI pipeline with 200 environment variables, half of which are undocumented
- A blue/green deployment that requires manual DNS cutover with a 15-minute propagation wait

---

## 21. Automation

**Scope:** Degree to which the system operates without human intervention — self-healing routines, automated testing, and CI/CD pipelines.

**Software examples:**
- A Kubernetes operator that automatically restarts failed pods and scales based on CPU
- A CI pipeline that runs 3000 tests and blocks merges on failure with no human gate
- Automated certificate rotation that eliminates the "expired cert at 3am" incident class
